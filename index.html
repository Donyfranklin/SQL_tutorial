Introduction

🗄️ Welcome to SQL

SQL (Structured Query Language) is how you talk to a database. Think of a database as a very organized Excel file — and SQL is the language you use to read, add, edit or delete data in it.

🗃️

Database

A collection of organized tables. Like an Excel workbook.

📋

Table

A grid of rows and columns. Like one Excel sheet.

📝

Row (Record)

One entry of data. Like one student's information.

🏷️

Column (Field)

A category of data. Like "Name" or "Age".

🔑

Primary Key

A unique ID for each row. No two rows share it.

🔗

Foreign Key

A column that links to another table's Primary Key.

📌
SQL is divided into 4 main categories
This guide covers DDL, DML, Clauses and Joins

DDL

🏗️ Data Definition Language

DDL commands define the structure of your database — like creating tables, modifying columns, or deleting tables. These are like the blueprint commands.

CREATE TABLE
Create a new table
Define columns and their data types
📋 Analogy: Like creating a new sheet in Excel and naming each column header.
CREATE TABLE table_name (
  column1 datatype [constraints],
  column2 datatype [constraints]
);
create_students_table.sql 
CREATE TABLE students (
  student_id   INT           PRIMARY KEY,
  name         VARCHAR(100) NOT NULL,
  age          INT           CHECK (age >= 16),
  email        VARCHAR(200) UNIQUE,
  city         VARCHAR(50)  DEFAULT 'Chennai',
  enrolled_on  DATE
);

ALTER TABLE
Modify an existing table
Add, remove or change columns
🔧 Analogy: Like adding a new column to an existing Excel sheet, or renaming a column header.

DROP / TRUNCATE
Delete a table or clear all its data
These are destructive — use with extreme care
drop_truncate.sql 
-- DROP: deletes the table completely (structure + data)
DROP TABLE students;

-- DROP IF EXISTS: safe version (no error if table doesn't exist)
DROP TABLE IF EXISTS students;

-- TRUNCATE: deletes ALL rows but keeps the table structure
TRUNCATE TABLE students;
CommandRemoves Data?Removes Table?Can Rollback?

⚠️ DROP removes the entire table forever. TRUNCATE removes all rows but keeps the table shell. Neither can be undone!

Constraints — Rules on Columns

CONSTRAINTS
Rules that protect your data quality
Applied when creating or altering a table
ConstraintWhat it meansExample
PRIMARY KEY Unique identifier for each row. Cannot be NULL. student_id INT PRIMARY KEY
NOT NULL This column must always have a value. name VARCHAR(100) NOT NULL
UNIQUE No two rows can have the same value here. email VARCHAR(200) UNIQUE
CHECK Value must pass a logical condition. age INT CHECK (age >= 16)
DEFAULT Uses this value if no value is provided. city VARCHAR(50) DEFAULT 'Chennai'
FOREIGN KEY Links to a Primary Key in another table. dept_id INT REFERENCES departments(id)
table_with_foreign_key.sql 
CREATE TABLE departments (
  dept_id   INT          PRIMARY KEY,
  dept_name VARCHAR(100) NOT NULL
);

CREATE TABLE employees (
  emp_id   INT          PRIMARY KEY,
  emp_name VARCHAR(100) NOT NULL,
  salary   DECIMAL(10,2) DEFAULT 30000.00,
  dept_id  INT,
  FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);
DML

✏️ Data Manipulation Language

DML commands interact with the actual data inside tables — reading, inserting, updating and deleting records.

SELECT
Read data from a table
The most frequently used SQL command
🔍 Analogy: Like using "Find" in Excel — you choose which rows and columns to see.

-- Select ALL columns from a table
SELECT * FROM students;

-- Select specific columns
SELECT name, age, city
FROM students;

-- Give a column an alias (rename for display)
SELECT name AS student_name, age AS student_age
FROM students;

-- Select with calculation
SELECT name, marks, (marks * 100 / 500) AS percentage
FROM students;

INSERT INTO
Add new rows to a table
One row or multiple rows at once
📝 Analogy: Like filling in a new student registration form and saving it to the spreadsheet.
insert_examples.sql 
-- Insert one row
INSERT INTO students (student_id, name, age, city)
VALUES (1, 'Priya', 21, 'Chennai');

-- Insert multiple rows at once
INSERT INTO students (student_id, name, age, city)
VALUES
  (2, 'Arjun', 22, 'Mumbai'),
  (3, 'Meera', 20, 'Delhi'),
  (4, 'Rahul', 23, 'Bangalore'),
  (5, 'Sneha', 21, 'Chennai');
💡 The order of values must exactly match the order of columns listed in parentheses.
UPDATE
Edit existing data in a table
Always use WHERE to target specific rows
🖊️ Analogy: Like finding Priya's row in Excel and changing her city from Chennai to Hyderabad.
update_examples.sql 
-- Update one column for a specific row
UPDATE students
SET   city = 'Hyderabad'
WHERE student_id = 1;

-- Update multiple columns at once
UPDATE students
SET   city = 'Pune', age = 24
WHERE name = 'Arjun';

-- Update all rows (no WHERE — use carefully!)
UPDATE students
SET   city = 'Unknown';  -- changes EVERY row!
⚠️ Always use WHERE with UPDATE! Without WHERE, every single row in the table gets changed.
DELETE
Remove rows from a table
Always use WHERE unless clearing all rows intentionally
🗑️ Analogy: Like selecting a row in Excel and pressing Delete to remove it.
delete_examples.sql 
-- Delete one specific row
DELETE FROM students
WHERE student_id = 3;

-- Delete rows matching a condition
DELETE FROM students
WHERE city = 'Delhi';

-- Delete ALL rows (keeps the table structure)
DELETE FROM students;
⚠️ Always use WHERE with DELETE! Without WHERE, every row is deleted permanently.
Clauses

🎛️ SQL Clauses

Clauses are added to your SELECT queries to filter, sort, group, and limit results. They're the power tools that make SQL so flexible.

WHERE
Filter rows by a condition
Like clicking the "Filter" button in Excel
where_clause.sql 
-- Simple condition
SELECT * FROM students
WHERE city = 'Chennai';

-- Number comparison
SELECT * FROM students
WHERE age > 20;

-- AND: both conditions must be true
SELECT * FROM students
WHERE city = 'Chennai' AND age >= 21;

-- OR: at least one condition must be true
SELECT * FROM students
WHERE city = 'Chennai' OR city = 'Mumbai';

-- NOT: negate a condition
SELECT * FROM students
WHERE NOT city = 'Delhi';

-- IN: match a list of values
SELECT * FROM students
WHERE city IN ('Chennai', 'Mumbai', 'Delhi');

-- BETWEEN: value in a range (inclusive)
SELECT * FROM students
WHERE age BETWEEN 18 AND 25;

-- LIKE: search by partial text (% is wildcard)
SELECT * FROM students
WHERE name LIKE 'P%';     -- starts with P

SELECT * FROM students
WHERE name LIKE '%a';     -- ends with a

SELECT * FROM students
WHERE name LIKE '%ri%';   -- contains 'ri'

-- IS NULL: find rows with no value
SELECT * FROM students
WHERE email IS NULL;

ORDER BY
Sort results in a specific order
ASC = A to Z (default) · DESC = Z to A
↕️ Analogy: Like clicking a column header in Excel to sort rows alphabetically or numerically.
order_by.sql 
-- Sort by name A → Z
SELECT * FROM students
ORDER BY name ASC;

-- Sort by age highest first
SELECT * FROM students
ORDER BY age DESC;

-- Sort by city first, then by name within each city
SELECT * FROM students
ORDER BY city ASC, name ASC;
DISTINCT
Remove duplicate values
Show each unique value only once
distinct.sql 
-- Get unique cities (no duplicates)
SELECT DISTINCT city FROM students;

-- Count unique cities
SELECT COUNT(DISTINCT city) AS unique_cities
FROM students;
LIMIT / OFFSET
Control how many rows to return
Great for pagination and previewing data
limit_offset.sql 
-- Show only the first 5 rows
SELECT * FROM students LIMIT 5;

-- Skip first 5, show next 5 (page 2)
SELECT * FROM students
LIMIT 5 OFFSET 5;

-- Top 3 oldest students
SELECT name, age
FROM students
ORDER BY age DESC
LIMIT 3;
GROUP BY
Group rows and calculate summaries
Like pivot tables in Excel
📊 Analogy: "How many students are from each city?" — GROUP BY bunches rows together so you can count, sum or average per group.
group_by.sql 
-- Count students from each city
SELECT city, COUNT(*) AS total_students
FROM   students
GROUP BY city;

-- Average age per city
SELECT city, AVG(age) AS avg_age
FROM   students
GROUP BY city
ORDER BY avg_age DESC;

HAVING
Filter groups after GROUP BY
WHERE filters rows · HAVING filters groups
🎯 Analogy: "Show me only cities that have MORE than 1 student." — HAVING is WHERE applied after grouping.
having_clause.sql 
-- Cities with more than 1 student
SELECT   city, COUNT(*) AS total
FROM     students
GROUP BY city
HAVING   COUNT(*) > 1;

-- Departments where avg salary > 50000
SELECT   dept_id, AVG(salary) AS avg_salary
FROM     employees
GROUP BY dept_id
HAVING   AVG(salary) > 50000;

Aggregate Functions

COUNT · SUM · AVG · MIN · MAX
Calculate totals, averages, and extremes
Used with SELECT and GROUP BY
aggregate_functions.sql 
-- COUNT: how many rows?
SELECT COUNT(*)        AS total_students   FROM students;
SELECT COUNT(email)    AS with_email       FROM students;  -- excludes NULLs

-- SUM: total of a numeric column
SELECT SUM(salary)    AS total_payroll    FROM employees;

-- AVG: average of a numeric column
SELECT AVG(salary)    AS average_salary   FROM employees;

-- MIN and MAX
SELECT MIN(age)       AS youngest         FROM students;
SELECT MAX(age)       AS oldest           FROM students;

-- Multiple aggregates in one query
SELECT
  COUNT(*)     AS total,
  AVG(salary)  AS avg_salary,
  MIN(salary)  AS min_salary,
  MAX(salary)  AS max_salary
FROM employees;

FULL OUTER JOIN
All rows from BOTH tables
Unmatched rows from either side show as NULL
🌐 Analogy: Show ALL students and ALL courses — match them where possible, fill NULL where there's no match on either side.
full_outer_join.sql 
SELECT
  s.name,
  c.course_name
FROM            students  AS s
FULL OUTER JOIN courses   AS c
            ON  s.student_id = c.student_id;

-- MySQL doesn't support FULL OUTER JOIN directly.
-- Workaround: LEFT JOIN UNION RIGHT JOIN
SELECT s.name, c.course_name
FROM students s LEFT JOIN courses c ON s.student_id=c.student_id
UNION
SELECT s.name, c.course_name
FROM students s RIGHT JOIN courses c ON s.student_id=c.student_id;
CROSS JOIN
Every row from left × every row from right
Creates all possible combinations
🎲 Analogy: You have 3 shirt sizes and 4 colors — CROSS JOIN gives you all 12 combinations (3×4).
cross_join.sql 
-- Every student paired with every course (useful for scheduling)
SELECT
  s.name,
  c.course_name
FROM        students  s
CROSS JOIN courses   c;
-- 4 students × 4 courses = 16 rows returned
SELF JOIN
A table joined with itself
Useful for hierarchical or comparative data
🪞 Analogy: An employee table where each employee has a "manager_id" that refers back to another employee in the same table.
self_join.sql 
-- employees table has: emp_id, name, manager_id
-- manager_id references emp_id in the SAME table

SELECT

  m.name       AS manager
FROM      employees  AS e

Putting It All Together — Full Example

🏆
A real-world query using everything you've learned
Students enrolled in courses, grouped by city, filtered by count
full_example_query.sql 
-- Find cities where more than 1 student is enrolled in a course,
-- show the city, number of enrolled students, and average age

SELECT
  s.city,
  COUNT(DISTINCT s.student_id) AS enrolled_students,
  AVG(s.age)                     AS average_age

FROM       students  AS s
INNER JOIN courses   AS c
       ON  s.student_id = c.student_id

WHERE    s.age BETWEEN 18 AND 30

GROUP BY s.city
HAVING   COUNT(DISTINCT s.student_id) > 1

ORDER BY enrolled_students DESC
LIMIT    10;
💡 This single query uses: SELECT, INNER JOIN, WHERE, BETWEEN, GROUP BY, HAVING, COUNT, AVG, DISTINCT, ORDER BY and LIMIT — all in one!
Normalization

🧹 Database Normalization

Normalization is the process of organizing a database to reduce redundancy and improve data integrity. It breaks large tables into smaller, related ones following a set of rules called Normal Forms.

📦

What is Redundancy?

Storing the same data in multiple places. Causes update anomalies and wastes space.

✂️

Decomposition

Splitting one big table into smaller related tables connected by keys.

🔗

Functional Dependency

Column A determines column B. E.g., StudentID → StudentName.

⚠️

Anomalies

Problems caused by bad design: Insert, Update and Delete anomalies.

Normal Forms Overview

FormRuleProblem it fixes
1NFEach column holds atomic (single) values. No repeating groups.Multi-valued columns (e.g., "Math, Science" in one cell)
2NFMust be in 1NF. Every non-key column depends on the entire primary key (no partial dependency).Partial dependency in composite keys
3NFMust be in 2NF. No non-key column depends on another non-key column (no transitive dependency).Transitive dependencies
BCNFStricter 3NF: every determinant must be a candidate key.Anomalies missed by 3NF in rare overlapping keys

1NF — First Normal Form

1NF
Atomic values — one fact per cell
No comma-separated lists, no repeating column groups
📋 Rule: Each column must contain a single, indivisible value. A cell cannot hold "Math, Science" — it must be split into separate rows.

2NF — Second Normal Form

2NF
No partial dependency on a composite key
Every non-key column must depend on the full primary key
📋 Applies when: Your primary key is made of two or more columns (composite key). A non-key column must depend on both key columns together, not just one.

3NF — Third Normal Form

3NF
No transitive dependency
Non-key columns must depend only on the primary key — not on each other
📋 Transitive dependency: A → B and B → C means A → C transitively. In 3NF, C must be moved to a separate table.
❌ Violates 3NF — ZipCode → City (non-key determines non-key)

3nf_example.sql 
-- Students table (no City column)
CREATE TABLE students (
  student_id  INT         PRIMARY KEY,
  name        VARCHAR(50),
  zip_code    VARCHAR(10) REFERENCES zip_codes(zip_code)
);

-- Separate table breaks the transitive dependency
CREATE TABLE zip_codes (
  zip_code  VARCHAR(10) PRIMARY KEY,
  city      VARCHAR(50)
);
💡 In practice, 3NF is the most commonly targeted form in production databases. BCNF is needed only in special edge cases with overlapping candidate keys.
Window Functions

🪟 Window Functions

Window functions perform calculations across a set of rows related to the current row — without collapsing rows like GROUP BY does. They use the OVER() clause to define the "window" of rows.

🏷️

OVER()

Defines the window. Required for all window functions. Can be empty to mean "all rows".

🗂️

PARTITION BY

Divides rows into groups (partitions) within the window. Like GROUP BY but rows are kept.

🔢

Ranking Functions

ROW_NUMBER, RANK, DENSE_RANK — assign a rank to each row within a partition.

↔️

LAG / LEAD

Access data from a previous or next row without a self-join.

OVER() and PARTITION BY

OVER / PARTITION BY
Define the window of rows
Compute totals per group while keeping all rows visible
📋 Key difference vs GROUP BY: GROUP BY returns one row per group. OVER() keeps every row and adds the group calculation as an extra column.
window_over.sql 
-- Total salary per department, shown alongside each employee
SELECT
  employee_name,
  department,
  salary,
  SUM(salary) OVER (PARTITION BY department) AS dept_total,
  AVG(salary) OVER (PARTITION BY department) AS dept_avg
FROM employees;

RANK FUNCTIONS
Assign sequential numbers or ranks to rows
Useful for top-N queries and leaderboards

SELECT
  employee_name,
  salary,

LAG and LEAD

LAG / LEAD
Access previous or next row values
Compare a row with its neighbors — no self-join needed
📋 LAG looks backward (previous row). LEAD looks forward (next row). The second argument is how many rows to look, and the third is the default if there's no such row.
lag_lead.sql 
-- Compare each month's sales to the previous month
SELECT
  month,
  sales,
  LAG(sales, 1, 0) OVER (ORDER BY month) AS prev_month_sales,
  LEAD(sales, 1, 0) OVER (ORDER BY month) AS next_month_sales,
  sales - LAG(sales, 1, 0) OVER (ORDER BY month) AS growth
FROM monthly_sales;

Transactions

🔒 Transactions

A transaction is a group of SQL statements that execute as one unit. Either all of them succeed, or none of them do — guaranteeing data consistency even when things go wrong.

⚛️

Atomicity

All operations succeed or all are rolled back. No partial updates.

✅

Consistency

Database moves from one valid state to another. Rules are never broken.

🔐

Isolation

Concurrent transactions don't interfere with each other.

💾

Durability

Once committed, changes survive power loss or crashes.

BEGIN, COMMIT, ROLLBACK

COMMIT / ROLLBACK
Save or undo a group of changes
The core of every transaction
🏦 Real-world analogy: A bank transfer — debit one account, credit another. If the credit fails, the debit must also be undone. COMMIT = confirm the transfer. ROLLBACK = cancel everything.
transaction_example.sql 
-- Start a transaction
BEGIN;

-- Debit account A
UPDATE accounts
SET    balance = balance - 1000
WHERE  account_id = 1;

-- Credit account B
UPDATE accounts
SET    balance = balance + 1000
WHERE  account_id = 2;

-- If everything is OK → save permanently
COMMIT;

-- If something went wrong → undo ALL changes
-- ROLLBACK;
⚠️ Without a transaction, each UPDATE auto-commits immediately. If the second UPDATE fails, the first is already saved — leaving the database in an inconsistent state.

SAVEPOINT

SAVEPOINT
Create a rollback point inside a transaction
Roll back to a specific point without undoing everything
🎮 Analogy: Like a save point in a video game. You can return to that exact moment without restarting from the very beginning.
savepoint_example.sql 
BEGIN;

INSERT INTO orders (order_id, product) VALUES (101, 'Laptop');

SAVEPOINT after_order;   -- ← bookmark this moment

INSERT INTO payments (order_id, amount) VALUES (101, 75000);

-- Oops — payment was wrong, undo only back to savepoint
ROLLBACK TO SAVEPOINT after_order;

-- Order row is still there; payment row is gone
INSERT INTO payments (order_id, amount) VALUES (101, 85000);

COMMIT;
💡 Use RELEASE SAVEPOINT name to remove a savepoint you no longer need, freeing resources.

DCL

🔑 Data Control Language — GRANT & REVOKE

DCL commands manage who can access what in your database. GRANT gives privileges to users or roles; REVOKE takes them away. Critical for database security.

👤

User / Role

A database user is an account. A role is a named collection of privileges you can assign to many users.

🛡️

Privilege

A specific permission: SELECT, INSERT, UPDATE, DELETE, CREATE, EXECUTE, ALL PRIVILEGES, etc.

🔑

WITH GRANT OPTION

Lets the recipient also grant this privilege to other users.

🚫

CASCADE

When revoking, CASCADE removes the privilege from users who received it via GRANT OPTION too.

GRANT

GRANT
Give a user or role specific privileges
Controls what actions a user can perform on database objects
GRANT privilege_list
ON object
TO user_or_role [WITH GRANT OPTION];
grant_examples.sql 
-- Allow a user to only read the students table
GRANT SELECT
ON    students
TO    'analyst_user';

-- Allow insert and update on orders
GRANT INSERT, UPDATE
ON    orders
TO    'sales_role';

-- Grant all privileges on a table
GRANT ALL PRIVILEGES
ON    products
TO    'admin_user';

-- Let the user also pass this privilege to others
GRANT SELECT
ON    reports
TO    'manager_user' WITH GRANT OPTION;

REVOKE

REVOKE
Remove previously granted privileges
Use CASCADE to also remove privileges passed on to others
REVOKE privilege_list
ON object
FROM user_or_role [CASCADE | RESTRICT];
revoke_examples.sql 
-- Remove SELECT privilege from the analyst
REVOKE SELECT
ON     students
FROM   'analyst_user';

-- Revoke INSERT and UPDATE from a role
REVOKE INSERT, UPDATE
ON     orders
FROM   'sales_role';

-- Cascade: also remove privileges the user passed to others
REVOKE SELECT
ON     reports
FROM   'manager_user' CASCADE;
⚠️ RESTRICT (default in some DBs) will fail if the user has already granted the privilege to someone else. Use CASCADE to remove it from everyone in the chain.
